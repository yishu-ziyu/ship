## 架构决策

> 任务 ID: 20260626-044005 | 阶段: 架构选型 | 日期: 2026-06-26
> 引用: 01-discovery.md（用户/竞品/机会判断）

### 架构指纹

- **形态**: 跨平台插件（Claude Code 插件 = 1 个可分发单元 + 13 个 skill + hooks + scripts），运行在用户本地 Claude Code 进程内
- **团队规模**: 单人开发 + 公开用户（GitHub 开源）—— 对应"独立开发者 / 小团队用 PM + 工程一体化工具"的用户画像（见 01-discovery.md 用户画像段）
- **用户技术栈**: 主要用户是"全栈工程师兼产品经理"（技术水平中高级，CLI 熟练），次要用户是非工程师 PM。**两类用户的技术敏感度差异大**——这是架构选择的最大约束
- **部署环境**: 本地安装（`/plugin install yishuship@local`）+ 跨平台（macOS/Linux/Windows）、跨 Claude Code 版本升级

### 候选架构

#### 候选 A：单一插件 + 13 skill + 路由脑（现状）

| 维度 | 说明 |
|------|------|
| 形态 | 1 个 plugin 仓（`~/.claude/plugins/yishuship/`），13 skill 通过路由脑按需加载 |
| 入口 | `/yishuship:use-yishuship` → 路由表分发到具体 skill |
| 状态 | `.ship/tasks/<task_id>/` 持久化，跨会话可恢复 |
| 训练 | SkillOpt 训练循环跑整个 plugin 的所有 skill 文档 |

#### 候选 B：多插件 + sub-agent 调度

| 维度 | 说明 |
|------|------|
| 形态 | 拆成 4-5 个独立 plugin（pm-intake、ship-core、visual-design、write-docs、arch-design），按需 `install` |
| 入口 | use-yishuship 路由表里调 `sub-agent`，sub-agent 在新 plugin 里执行 |
| 状态 | 跨 plugin 共享 `.ship/tasks/<task_id>/`（目录约定，不强制） |
| 训练 | 每个 plugin 独立训练，SkillOpt 数据需跨 plugin 归一化 |

#### 候选 C：极简拆分（PM 内嵌 + 6 原子 skill）

| 维度 | 说明 |
|------|------|
| 形态 | 删 pm-intake 独立 skill，PM 流程作为 use-yishuship 路由脑的内置流程；只留 6 个原子工程 skill（design/dev/review/qa/e2e/handoff） |
| 入口 | `/yishuship:use-yishuship` 既是路由脑也是 PM 流程 |
| 状态 | 同 A |
| 训练 | SkillOpt 跑 use-yishuship 单 skill，PM 阶段不能独立评分 |

### 关键差异点

| 维度 | A（现状）| B（多插件）| C（极简）|
|------|---------|-----------|----------|
| 装一次能用全部能力 | ✅ | ❌ 需装 4-5 次 | ✅ |
| PM 阶段独立训练 | ✅ pm-intake 独立 | ✅ | ❌ PM 评分消失 |
| 跨 skill 共享状态 | ✅ 同 plugin | ❌ 需约定 | ✅ |
| 安装包大小 | 大（一次到位）| 小（按需）| 中 |
| 对非工程师用户的认知负担 | 低（命令都带 `yishuship:` 前缀）| 高（要懂 plugin 边界）| 最低（PM 直接嵌入）|
| SkillOpt 训练数据量 | 高（13 skill 一起）| 低（单 plugin）| 极低（1 skill）|

### 决策记录

**选定架构**：**候选 A — 单一插件 + 多 skill（现状）**

**选择理由**：
1. **用户技术栈的差异**：两类用户（工程师 + 非工程师 PM）共享同一命令空间 `yishuship:*`，**避免非工程师用户在多个 plugin 之间切换**——这是非工程师用户的最大摩擦点
2. **PM 阶段独立优化是核心壁垒**：63 维度评分 + SkillOpt 训练循环是 yishuship 跟原版 Ship 的唯一差异化（见 01-discovery.md「我们的优势」段）。候选 C 把 PM 嵌入路由脑会丢掉这个独立性，候选 B 拆插件后 PM 评分数据要跨 plugin 归一化，工程复杂度陡增
3. **跨 skill 状态共享**：`.ship/tasks/<task_id>/` 在同 plugin 内天然共享，候选 B 需要约定目录路径，违反「文件系统是隐式 API 的接口要少」原则
4. **状态机一致性**：原版 Ship 提供的 4 个 hook（phase-guardrail / pm-gate / stop-gate / pm-verify）按 plugin 粒度注册，拆插件后 hook 边界要重画

### 替代方案

| 替代 | 用户拒绝理由 | 我们看到的潜在代价 |
|------|------------|------------------|
| 候选 B（多插件）| 安装摩擦大、对非工程师用户不友好、PM 训练数据难归一化 | sub-agent 调度延迟；状态共享约定变成隐性技术债 |
| 候选 C（极简）| PM 流程不能独立评分 → SkillOpt 的 PM 训练循环无法运行 → 失去最大差异化 | 63 维度评分框架作废；Step 1.5 架构选型子阶段无处安放 |

**架构指纹 → 架构选择的因果链**：

- 形态 = 跨平台插件 + 本地安装 → 候选池收敛到「单仓多 skill」vs「多仓多 plugin」vs「单 skill 极简」（A/B/C）
- 团队规模 = 单人开发 + 公开用户 → 排除「微服务 / 独立部署」类架构（运维成本与用户不匹配）
- 用户技术栈 = 工程师 + 非工程师混合 → 倾向 A（统一命令空间降低非工程师用户摩擦）
- 部署环境 = 本地安装 + 跨平台 → 排除「Serverless / 云端」类架构（用户必须本地跑 Claude Code）

**对后续阶段的约束**：

- **Step 2 定义**：
  - 「Non-goals」必须包含「不做多插件拆分」「不做 PM 层独立成产品」
  - 「范围封顶」的 1 个原子不能做（拆插件）
- **Step 3 设计**：
  - 「技术方案」必须基于「Claude Code Plugin 规范 + Skill + Hook + Script」四件套，不要引入新运行时（如 docker、browser）
  - 「API 设计」适用于 plugin 内部 skill 之间的接口（marketplace.json 描述符、文件约定），不是 HTTP API
- **Step 4 验证**：
  - 「核心假设」必须包含「Claude Code 插件规范在 1-2 年内不会大变」（架构风险）
  - 「Plan B」需要为这个假设设计：若 plugin 规范大改，yishuship 可以降级为「Claude Code Skill 集合 + 用户手动调用」，不依赖 marketplace.json

### 边界（什么时候重新评估这个决策）

- Claude Code 官方宣布 plugin 规范废弃 → 立刻重评
- 13 skill 增加到 25+ skill，单 plugin 安装包超过 50MB → 重新评估「按需装」（候选 B 局部重启）
- SkillOpt 训练循环证明对 PM 评分无增益 → 可以考虑候选 C（PM 嵌入路由脑）
